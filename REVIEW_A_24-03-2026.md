# Review A - 24-03-2026

## Findings

1. **High**: Gateway hot reload does not reapply logging configuration, so changes to `logging.level`, `logging.console.level`, and `logging.file_logging.*` are ignored until a full process restart. Startup applies logger config once in [`pkg/gateway/gateway.go:91`](/home/openclaw/picoclaw/pkg/gateway/gateway.go#L91), but the hot-reload path in [`pkg/gateway/gateway.go:439`](/home/openclaw/picoclaw/pkg/gateway/gateway.go#L439) rebuilds providers and services without calling `logger.ApplyConfig` again. That breaks the advertised configurable logging behavior for the long-lived gateway process.

2. **Medium**: In-process logger reconfiguration cannot disable file logging once it has been enabled. [`logger.ApplyConfig`](/home/openclaw/picoclaw/pkg/logger/config.go#L23) only enables the file sink when `cfg.FileLogging.Enabled` is true, and there is no `else` branch to call `DisableFileLogging` when it becomes false; see [`pkg/logger/config.go:53`](/home/openclaw/picoclaw/pkg/logger/config.go#L53). Because the logger is package-global state, any future reload or restart-in-process path will keep writing to the old file unexpectedly.

3. **Medium**: `--debug` currently short-circuits the rest of logger setup, which means debug mode silently skips configured file logging and any explicit console and file level split. The early return in [`pkg/logger/config.go:24`](/home/openclaw/picoclaw/pkg/logger/config.go#L24) calls only `SetLevel(DEBUG)` and exits before sink configuration. In practice, starting the gateway or CLI with debug enabled gives different outputs, not just a more verbose level.
